# Library Book Borrowing — Hexagonal Architecture

## Опис use case

Система видачі книг у бібліотеці реалізує сценарій, у якому студент бере книгу на читання.

Система виконує такі кроки:
- перевіряє, чи існує книга з вказаним ID;
- перевіряє, чи існує студент з вказаним ID;
- перевіряє бізнес-правило: книга має мати вільні екземпляри;
- перевіряє бізнес-правило: студент не може мати більше 3 книг одночасно;
- зменшує кількість доступних екземплярів книги;
- фіксує книгу у списку студента;
- створює запис про видачу (Loan) і зберігає його.

## Обрана архітектура

**Hexagonal Architecture (Ports and Adapters)**

## Обґрунтування вибору

Hexagonal Architecture ідеально підходить для цього завдання, тому що:

- бізнес-логіка (перевірка доступності, ліміт книг) є ядром системи і не повинна залежати від того, де зберігаються дані — в пам'яті, файлі чи базі даних;
- порти (інтерфейси) чітко визначають, що система очікує від зовнішнього світу;
- адаптери можна легко замінити: наприклад, `InMemoryBookRepository` можна замінити на `PostgresBookRepository` без змін у бізнес-логіці;
- use case залишається чистим і тестованим незалежно від інфраструктури.

## Структура проєкту

```
src/
├── domain/               # Доменні сутності (ядро системи)
│   ├── Book.ts
│   ├── Student.ts
│   └── Loan.ts
├── application/          # Use case — сценарій використання
│   └── BorrowBookUseCase.ts
├── ports/                # Інтерфейси (порти) — межа між ядром і зовнішнім світом
│   ├── BookRepository.ts
│   ├── StudentRepository.ts
│   └── LoanRepository.ts
├── adapters/             # Адаптери — технічні реалізації портів
│   ├── InMemoryBookRepository.ts
│   ├── InMemoryStudentRepository.ts
│   └── InMemoryLoanRepository.ts
└── main.ts               # Точка входу — збирає систему та запускає use case
```

## Розподіл відповідальностей

### Domain (`src/domain/`)
Містить бізнес-логіку і бізнес-правила. Не залежить ні від чого зовнішнього.

- **`Book.ts`** — перевіряє доступність екземплярів (`isAvailable()`), зменшує їх кількість (`borrowCopy()`), кидає помилку якщо книга недоступна.
- **`Student.ts`** — зберігає список взятих книг, перевіряє ліміт у 3 книги (`canBorrow()`), кидає помилку при перевищенні.
- **`Loan.ts`** — запис про видачу книги: ID, bookId, studentId, дата створення.

### Application (`src/application/`)
Містить сценарій використання — оркеструє доменні об'єкти і викликає порти.

- **`BorrowBookUseCase.ts`** — отримує книгу і студента через порти, викликає доменну логіку, зберігає результат. Не знає нічого про те, де зберігаються дані.

### Ports (`src/ports/`)
Інтерфейси, які визначають, що бізнес-логіка очікує від зовнішнього світу. Це межа між ядром і інфраструктурою.

- **`BookRepository.ts`** — `findById`, `save`
- **`StudentRepository.ts`** — `findById`, `save`
- **`LoanRepository.ts`** — `save`, `findAll`

### Adapters (`src/adapters/`)
Технічні реалізації портів. У цьому проєкті — зберігання в пам'яті. Можна замінити на будь-яку іншу реалізацію без змін у бізнес-логіці.

- **`InMemoryBookRepository.ts`**
- **`InMemoryStudentRepository.ts`**
- **`InMemoryLoanRepository.ts`**

### Main (`src/main.ts`)
Точка входу. Створює адаптери, підключає їх до use case через порти, запускає сценарій і демонструє перевірку бізнес-правил.

## Бізнес-правила

1. Книга повинна мати щонайменше один вільний екземпляр — інакше видача неможлива.
2. Студент не може мати більше 3 книг одночасно — інакше видача неможлива.
